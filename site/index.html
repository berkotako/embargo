Embargo sits in front of the npm registry and refuses to serve package versions until they pass age, provenance, and behavioral-risk checks. A firewall that blocks — not a scanner that warns after the fact.

The problem

The supply chain is the attack surface

The npm ecosystem has been hit by a wave of supply-chain attacks — token-hijacked releases of hugely popular packages, self-propagating worms, review-bypass via orphan commits, and install- or import-time backdoors. Native package-manager cooldowns help against the smash-and-grab window, but they're global, have no exception mechanism, and can't act on why a version is suspicious. Embargo adds what the package managers structurally won't: per-scope policy, fast-track exceptions, signal gating, CI admission control, and install containment.

The model

Every version gets one verdict

Embargo intercepts the package metadata your resolver fetches and filters it — so npm, pnpm, Yarn, and Bun simply never pick a held or denied version.

ALLOW
served normally
HOLD
withheld during cooldown / review
DENY
blocked — flagged by a signal or advisory
How it works

One line in .npmrc

Point your client at Embargo. Held and denied versions are stripped before your resolver ever sees them — a pinned-but-held version degrades to a clear error with an approval link, never a cryptic resolver failure.

Architecture

The engine is the brain

A client points registry= at the gateway, which asks the engine for verdicts over mutual-TLS and strips disallowed versions. The engine resolves cooldown + per-scope policy + provenance + behavioral signals + advisories — escalating HOLD→DENY permanently when a version is flagged mid-cooldown.

Defense in depth

Three enforcement layers, one engine

A miss at one layer is caught at the next — all fed by the same policy & signal engine.

L1 · Gateway

Resolution-time firewall

A Verdaccio plugin rewrites the packument, stripping HOLD/DENY versions. Catches smash-and-grab token-hijack releases, missing provenance, and republish anomalies.

L2 · Admission

CI gate

A CLI + GitHub Action fails any build whose lockfile diff introduces a held or denied version — diff-aware, so CI stays fast. Where most attacks actually land.

L3 · Sandbox

Install containment

A namespaced, egress-allowlisted install runner blocks and captures phone-home postinstalls and the runtime secret→egress chain.

Core · Engine

Policy & signal engine

Rust. Most-specific-wins per-scope policy, cooldown, provenance, and behavioral signals scored as chains — never single APIs — with OSV advisory matching.

UI · Console

Review & approvals

Quarantine review, policy authoring, time-boxed approvals, and a tamper-evident audit log — OIDC login with server-side RBAC.

Policy

Policy as code

A versioned, per-scope YAML DSL: hold public packages, pass @org/* instantly, fast-track emergency CVE fixes.

Quick start

Up in one command

Brings up the whole stack (engine, gateway, console, Postgres, Redis), waits for health, and prints how to point a client at it.

# clone, then bring the stack up
git clone https://github.com/berkotako/embargo && cd embargo
make up            # or: docker compose up --build

# firewall a project with one line
cd my-project
echo 'registry=http://localhost:4873/' >> .npmrc
npm install         # held / denied versions are stripped before resolve

Deploy a released build from published images, or read the design — it's all in the docs.
